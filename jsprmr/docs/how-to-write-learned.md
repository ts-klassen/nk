# `learned/NN_*.md` の書き方

このドキュメントは、JSPrimer の各ページで「学ぶこと」を `learned/NN_*.md` に書くための方針を定めます。

`learned/NN_*.md` は、課題を作るときの知識範囲の契約です。たとえば 15 の課題は、`learned/15_*.md` を正しく理解していないと解けない内容にし、`learned/01_*.md` から `learned/14_*.md` までの知識はすでにある前提にできます。一方で、`learned/16_*.md` 以降の知識や、教材でまだ扱っていない外部知識を要求してはいけません。

## 目的

- そのページを読んだ学習者に要求してよい知識を明確にする
- 課題が先の章の知識を暗黙に要求しないようにする
- 課題作成者が、教材本文ではなく `learned/NN_*.md` を見て要求知識を判断できるようにする

## 入力と出力

- 入力: `upstream/c1/NN_slug` の教材本文
- 出力: `learned/NN_slug.md`
- ファイル名の `NN_slug` は、原則として `upstream/c1/NN_slug` と揃える

## 基本方針

### 教材本文に忠実に書く

`learned/NN_*.md` には、そのページで説明されている知識だけを書く。一般的な JavaScript の知識であっても、そのページまたは以前の `learned` ファイルに出てこないものは含めない。

教材本文の例がある場合は、例の丸写しではなく「その例から要求できる判断・操作」として書く。

### 課題で検査可能な形にする

単なる見出しや用語集ではなく、課題で要求できる行動として書く。

よい例:

- `const` で再代入できない変数を宣言できる
- `Array.prototype.map` が元の配列を変更せず、新しい配列を返すことを説明できる
- `Promise` の fulfilled/rejected の状態に応じて `then`/`catch` で処理を分けられる

避ける例:

- `const`
- `map について`
- `Promise を理解する`

### 粒度は「課題の要求条件」に合わせる

1 つの箇条書きは、課題の採点基準に使える程度の大きさにする。細かすぎる構文断片や、複数の独立した知識を詰め込んだ文は避ける。

目安:

- 1 項目は 1 つの概念、構文、メソッド、または判断基準を扱う
- 同じ見出し内で強く関連するものはまとめてよい
- 似たメソッドでも、振る舞いや使いどころが異なるなら分ける

### 既習知識との境界を意識する

`learned/NN_*.md` は、そのページで新しく要求可能になる知識を書く。前の章ですでに扱った基礎知識を、毎回すべて書き直さない。

ただし、そのページの内容を理解するために前提となる既習知識が重要な場合は、`前提として使う既習知識` に短く書いてよい。

### 先の章の知識を混ぜない

教材本文中に名前だけ出てくるが、詳しい説明が先の章にある内容は、そのページの要求知識として扱わない。課題で使う場合も、実装に必須の知識として要求しない。

たとえば、あるページで「あとで説明する」とされている概念は、`learned` には「名前を見たことがある」程度に留めるか、書かない。

### 環境依存の知識を分ける

Node.js、ブラウザ、開発者ツール、DOM、ネットワーク、パッケージ管理など、JavaScript の文法そのものではない知識は、教材本文でそのページの主題として扱われている場合だけ要求知識に含める。

課題が実行環境を必要とする場合は、`課題で扱う場合の注意` に明記する。

## 標準構成

各 `learned/NN_*.md` は、次の構成を基本とする。

```markdown
# NN. ページタイトル

## このページで学ぶこと

- ...
- ...

## 課題で要求してよい知識

- ...
- ...

## 前提として使う既習知識

- ...

## 課題で要求しない知識（必要な場合のみ）

- ...

## 課題で扱う場合の注意

- ...
```

### `このページで学ぶこと`

ページ全体の要約を書く。教材の見出し構成に沿って、何を扱うページなのかを短く整理する。

### `課題で要求してよい知識`

課題作成で最も重要なセクション。学習者に実装・説明・修正・読解として要求できる知識を書く。

書き方:

- `...できる`
- `...を説明できる`
- `...を区別できる`
- `...の結果を予測できる`
- `...を使って...できる`

### `前提として使う既習知識`

そのページの課題で自然に使う、前章までの代表的な知識を書く。網羅しすぎず、課題の前提として重要なものに絞る。

例:

- 変数宣言、条件分岐、関数定義を使える
- オブジェクトのプロパティを読み書きできる
- 配列を作成し、要素へアクセスできる

### `課題で要求しない知識（必要な場合のみ）`

課題に混入しやすいが、このページではまだ要求できない知識がある場合だけ書く。網羅する必要はない。特に、次章以降で扱う知識、教材本文で深掘りしていない関連 API、環境固有の知識などに絞る。

書くことがなければ、このセクションは省略してよい。

例:

- DOM API を使った画面操作は要求しない
- 正規表現の高度な先読み・後読みは要求しない
- モジュール分割やビルドツールの知識は要求しない

### `課題で扱う場合の注意`

課題化するときの制約を書く。教材本文の説明だけでは誤解されやすい点や、採点時に揺れやすい点を明確にする。

例:

- 複数の解法が自然に成立する場合は、特定のメソッドに固定するかを課題文で明示する
- 破壊的な操作を許可するかどうかを課題文で明示する
- エラー文の完全一致ではなく、エラーの種類や原因の説明を評価する

## 書く手順

1. `upstream/c1/NN_slug` の `title`、`description`、本文見出しを確認する
2. 見出しごとに、その節で新しく扱われる知識を抜き出す
3. 抜き出した知識を「課題で要求できる行動」に言い換える
4. 前章までの知識と重複する項目を削る
5. 次章以降や本文外の知識を要求していないか確認する
6. 必要に応じて `課題で要求しない知識` と `課題で扱う場合の注意` を補う

## 判断基準

迷った場合は、次の基準で判断する。

- 教材本文の説明だけで学習者が到達できるなら書く
- 本文のコード例を少し変えた問題に対応できるなら書く
- MDN や仕様書を追加で読まないと解けないなら、原則として書かない
- 実務上は重要でも、教材のその時点で未説明なら書かない
- 以前の章で説明済みなら、この章の新規知識としては書かない

## 品質チェック

作成後に次を確認する。

- `課題で要求してよい知識` の各項目が、課題の要求条件として使える文になっている
- 後続章の知識を前提にしていない
- 「理解する」「知る」だけの曖昧な項目が残っていない
- 教材本文の見出しに対して、大きな抜け漏れがない
- 課題で禁止・注意すべき範囲がある場合は明示されている
