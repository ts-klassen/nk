# 採点システム仕様

このドキュメントは、課題の採点、テスト実行、ソースコード保存、バックエンド API の仕様を定める。

採点はフロントエンド側で実行する。バックエンドは HTML/CSS/JavaScript のホスティング、Markdown の問題文提供、テストケース提供、課題セット一覧提供、ソースコードのアップロード保存、最新保存コードの取得を担当する。

## 実装方針

実装は ESM で行う。`package.json` には `"type": "module"` を指定し、CommonJS は使わない。

依存は必要最低限にする。バックエンドのために追加する外部依存は原則として Express.js だけにし、Markdown 変換、バリデーション、テスト補助などのために追加パッケージを増やさない。

起動は npm script に依存しない簡単な方式にする。

```sh
node src/server.js --submissions-dir submissions
```

バックエンド起動時に提出コード保存先ディレクトリを指定しない場合は起動しない。

コードの自動テストは用意しない。実装時に確認が必要な場合は、手動でサーバを起動し、ブラウザまたは `curl` で最低限の動作確認を行う。

実装で迷う場合は、機能、依存、設定、抽象化を増やさず、シンプルな方を選ぶ。

## 点数

1 つの課題セットは、次の 3 設問で構成される。

| 設問 | 点数 |
| --- | ---: |
| 基本 | 4 |
| 標準 | 2 |
| 応用 | 1 |

満点は 7 点とする。

各設問は、その設問に用意されたすべてのテストケースを通した場合だけ点数を与える。1 つでも失敗したテストケースがある場合、その設問は 0 点とする。

例:

- 基本、標準、応用のすべてに正解: 7 点
- 基本と応用に正解、標準に不正解: 5 点
- 標準だけ正解: 2 点
- すべて不正解: 0 点

## 実行モデル

フロントエンドは、各テストケースごとに提出コードを実行する。

Worker は使わない。採点はフロントエンドの通常の JavaScript で、選択中の設問のテストケースを 1 件ずつ順番に実行する。

各テストケースの `input` は、次の 3 種類のいずれかで生成する。

- `input.type: "value"`: `input.value` を JSON 値として複製し、`input` を生成する
- `input.type: "constructor"`: `input.name` のコンストラクタを `new` し、`input.args` を渡して `input` を生成する
- `input.type: "factory"`: `input.factory` の関数を実行し、その戻り値を `input` として使う

例:

```json
{
  "input": {
    "type": "constructor",
    "name": "Date",
    "args": ["2020-01-02T00:00:00.000Z"]
  }
}
```

この場合、提出コードには次のように生成された値が渡る。

```js
const input = new Date("2020-01-02T00:00:00.000Z");
```

`input.type: "value"` の `input.value` と、`input.type: "constructor"` の `input.args` は JSON で表現できる値にする。`input.args` を省略した場合は空配列として扱う。

`input.type: "constructor"` で使用できるコンストラクタは、フロントエンドが明示的に許可したものに限る。許可されていないコンストラクタ名、存在しないコンストラクタ名、またはコンストラクタではない値が指定された場合、そのテストケースは問題定義エラーとして扱う。

`input.type: "factory"` の `input.factory` は、JavaScript の関数式またはアロー関数式の文字列とする。フロントエンドはこれを関数として評価し、引数なしで呼び出した戻り値を `input` にする。

```js
const input = inputFactory();
```

ファクトリが関数として評価できない場合、戻り値を生成できない場合、または例外を投げた場合、そのテストケースは問題定義エラーとして扱う。

`input.type: "factory"` は任意の JavaScript オブジェクトを作れるため、問題ファイルが信頼済みである前提でだけ使う。外部ユーザーが投稿したテストケースや、信頼できない JSON から `input.factory` を実行してはいけない。

テスト実行時には、提出コードの前に次の変数が定義されている。

```js
const input = ...;
let output;
```

提出コードは `input` を読み、`output` に結果を代入する。

提出コードは `Function` コンストラクタで実行する。採点用の Worker ファイルや複雑な隔離機構は追加しない。

```js
const execute = new Function("providedInput", `
  const input = providedInput;
  let output;
  ${code}
  return output;
`);
const output = execute(input);
```

コード入力画面では、`textarea` の上部に次のような実行前提を表示する。

```js
const input = テストケースごとの値;
let output;
```

学習者が `const input` や `let output` を自分で書く必要はない。

## 正誤判定

各テストケースは、`output.type: "value"` または `output.type: "evaluator"` のどちらかで判定する。

`output.type: "value"` の場合は、実行後の `output` と、テストケース定義の `output.value` を比較する。

`output.type: "evaluator"` の場合は、JSON に文字列として保存された `output.evaluator` をフロントエンドで関数として評価し、実行後の `output` と元の `input` を渡す。評価関数が `true` を返した場合だけ、そのテストケースを成功とする。

```js
const passed = outputEvaluator(output, originalInput) === true;
```

評価関数が `false` を返した場合、真偽値以外を返した場合、例外を投げた場合、または関数として評価できない場合、そのテストケースは失敗とする。

提出コードが `input` の中身を変更しても評価関数の判定が変わらないように、評価関数へ渡す `input` は提出コード実行前の値を使う。実装では、提出コード用の `input` と評価関数用の `originalInput` を同じ入力指定から別々に生成する。`input.type: "factory"` の場合も、同じ `input.factory` を 2 回呼び出して別々の値を生成する。

`output.type: "value"` の比較は JSON で表現できる値を対象にする。

推奨する比較ルール:

- 文字列、数値、真偽値、`null` は `Object.is` 相当で比較する
- 配列は、長さと各要素を再帰的に比較する
- オブジェクトは、キー集合と各値を再帰的に比較する
- オブジェクトのキー順は一致条件に含めない

`undefined`、関数、`NaN`、`Infinity`、`-Infinity` は `input.value`、`input.args`、`output.value` に使わない。`Date`、`Map`、`Set` など JSON で直接表せない入力は、`input.type: "constructor"` で生成する。`Date`、`Map`、`Set` など JSON で直接表せない期待出力を判定したい場合は、`output.type: "evaluator"` を使う。

提出コードが例外を投げた場合、そのテストケースは失敗とする。

Worker は使わないため、提出コードの強制停止やタイムアウトは実装しない。

## テストケースの開示

テストケースはすべてフロントエンドで表示できるようにする。

学習者は、各テストケースの `input` 生成指定と `output` 判定指定を確認できる。隠しテストは設けない。

## ソースコード保存

保存ボタンを押したとき、またはテストを実行したときに、入力されたコードをバックエンドへ送信し、ファイルへ保存する。

保存は採点結果に関係なく行う。バックエンドは保存だけを行い、正誤判定や点数計算は行わない。

保存ボタンは、現在の `textarea` の内容と、最後に保存または読み込んだコードに差分がある場合だけ有効にする。差分がない場合は無効化する。

最新保存コードの読み込み中、および保存 API の実行中は、`textarea` を無効化する。保存が完了したら、テスト実行中であっても `textarea` は再び有効にする。

未保存の差分がある状態で単元、課題セット、設問を切り替えようとした場合は、保存せずに切り替えるか確認する。キャンセルした場合は切り替えず、現在の入力内容を残す。保存せずに進むことを選んだ場合は、未保存の入力内容を破棄して切り替える。

未保存の差分がある状態でページを離脱または再読み込みしようとした場合は、ブラウザ標準の離脱確認を表示する。利用者は警告を無視して保存せずに離脱できる。

テスト実行時は、差分の有無に関係なく必ず保存 API を呼び出す。すでに保存済みのコードであっても、テスト実行時点のコードを新しい提出として保存する。保存が成功したら、そのコードを最新の保存済みコードとして扱い、保存ボタンを無効化する。

保存先ディレクトリは、バックエンド起動時のオプションで指定する。オプションが未指定の場合、バックエンドは起動しない。

指定されたディレクトリが存在しない場合は作成する。権限不足などで作成または保存できない場合は、HTTP 500 を返す。

保存パスは次の形式にする。

```text
${dir}/${unit_number}/${set_index}/{basic,standard,advanced}-${timestamp}.js
```

例:

```text
submissions/03/1/basic-1710000000.js
```

`timestamp` は Unix 秒とする。

同じ保存先ファイルがすでに存在する場合、バックエンドは HTTP 429 Too Many Requests を返す。429 を受け取ったフロントエンドは、1 秒後に同じ内容を再送する。

ページ表示時、単元変更時、課題セット変更時、難易度変更時は、選択中の単元、課題セット、難易度に対応する最新保存コードをバックエンドから取得して `textarea` に入れる。最新保存コード API が 404 を返した場合は、`textarea` を空欄にする。

最新保存コードは、ファイル名内の `timestamp` が最も大きいものを返す。

## バリデーション

バックエンドは URI から受け取る `unit_number` と `set_index` を必ずバリデーションする。

`unit_number`:

- `03` から `30` までの 2 桁文字列だけを受け付ける
- パス区切り文字や `..` を含む値は受け付けない

`set_index`:

- `1` 以上の整数文字列だけを受け付ける
- 先頭ゼロは受け付けない
- パス区切り文字や `..` を含む値は受け付けない

`difficulty`:

- `basic`、`standard`、`advanced` のいずれかだけを受け付ける

不正な値を受け取った場合は HTTP 400 を返す。

## バックエンド API

Express.js で実装する。サーバのエントリポイントは `src/server.js` とする。

### 静的ファイル

HTML/CSS/JavaScript を配信する。

単元名は `public/units.json` に JSON として配置する。フロントエンドはこの JSON を読み込み、単元プルダウンや現在の読み込み状態に単元番号と単元名を表示する。

### 課題セット一覧取得

指定された単元に存在する課題セット番号の一覧を返す。

例:

```http
GET /api/problem-sets/03
```

レスポンス:

```json
{
  "sets": ["1", "2"]
}
```

- 成功時: `200 OK`、`application/json`
- 単元が存在しない場合: `200 OK`、空の `sets`
- パラメータ不正: `400 Bad Request`
- 一覧取得失敗: `500 Internal Server Error`

### 問題文取得

指定された課題文 Markdown を返す。

例:

```http
GET /api/problems/03/1/basic.md
```

レスポンス:

- 成功時: `200 OK`、`text/markdown`
- 対象がない場合: `404 Not Found`
- パラメータ不正: `400 Bad Request`

### テストケース取得

指定されたテストケース JSON を返す。

例:

```http
GET /api/problems/03/1/basic.json
```

レスポンス:

- 成功時: `200 OK`、`application/json`
- 対象がない場合: `404 Not Found`
- パラメータ不正: `400 Bad Request`

### ソースコードアップロード

保存ボタンまたはテスト実行時のコードを保存する。

例:

```http
POST /api/submissions/03/1/basic
Content-Type: application/json

{
  "code": "output = input + 1;"
}
```

レスポンス:

- 保存成功: `201 Created`
- 同じ Unix 秒のファイルがすでに存在: `429 Too Many Requests`
- パラメータ不正: `400 Bad Request`
- 保存失敗: `500 Internal Server Error`

`code` は文字列だけを受け付ける。空文字列も許可し、ファイル保存を行う。

### 最新ソースコード取得

選択中の単元、課題セット、難易度に対応する提出コードのうち、ファイル名内のタイムスタンプが最も新しいものを返す。

例:

```http
GET /api/submissions/03/1/basic/latest
```

レスポンス:

```json
{
  "code": "output = input + 1;",
  "filename": "basic-1710000000.js",
  "timestamp": 1710000000
}
```

- 成功時: `200 OK`、`application/json`
- 対象がない場合: `404 Not Found`
- パラメータ不正: `400 Bad Request`
- 読み込み失敗: `500 Internal Server Error`

## フロントエンド

フロントエンドは次を行う。

- 単元、課題セット、設問を選択できる UI を表示する
- 単元は `public/units.json` の単元名を使い、単元番号と単元名を合わせて表示する
- 課題セットは「課題セット」と表記し、バックエンドから取得した一覧をプルダウンで表示する
- 読み込みボタンは表示せず、単元、課題セット、設問の変更を即座に反映する
- Markdown の課題文は HTML 変換せず、テキストとして表示する
- 開示されたテストケースを表示する
- `input.type: "constructor"` のテストケースでは、生成される `new Constructor(...args)` の形を表示する
- `input.type: "factory"` のテストケースでは、ファクトリ関数のソース文字列を表示する
- `output.type: "evaluator"` のテストケースでは、評価関数のソース文字列を表示する
- `textarea` の上部に `const input = ...;` と `let output;` の前提を表示する
- ページ表示時、単元、課題セット、設問の変更時に最新保存コードを取得して `textarea` に表示する
- 最新保存コードが 404 の場合は `textarea` を空欄にする
- テスト実行ボタンの横に保存ボタンを表示する
- 保存ボタンは差分がある場合だけ有効にし、押されたらコードをバックエンドへ POST して保存する
- 最新保存コードの読み込み中と保存 API の実行中は `textarea` を無効化する
- 未保存の差分がある状態で単元、課題セット、設問を切り替えようとした場合は、保存せずに切り替えるか確認する
- 未保存の差分がある状態でページ離脱または再読み込みをしようとした場合は、ブラウザ標準の離脱確認を表示する
- テスト実行時にコードをバックエンドへ POST して保存する
- テスト実行時の保存は差分の有無に関係なく必ず行う
- 保存 API が 429 を返した場合、1 秒後に再送する
- 保存 API が 500 を返した場合、保存失敗として表示する
- 各設問の全テストケース結果から、4 点、2 点、1 点を合算して表示する

採点はバックエンドへ送らない。バックエンドは採点結果を信頼したり保存したりしない。

## 品質チェック

実装前に次を確認する。

- 課題ファイルの配置規則が `problems/${unit_number}/${set_index}/...` に統一されている
- バックエンドが ESM で実装され、`node src/server.js --submissions-dir ...` で起動できる
- 外部依存が必要最低限である
- コードの自動テストを追加していない
- `unit_number`、`set_index`、`difficulty` のバリデーション方針が明確である
- フロントエンド実行時の入力変数と出力変数が `input`、`output` に統一されている
- Worker を使わず、提出コードを `Function` コンストラクタで実行する
- 提出コードのタイムアウトを実装していない
- テストケースの `input.type` ごとに、`value`、`constructor`、`factory` のどれで入力を生成するかが明確である
- コンストラクタ入力は許可されたコンストラクタだけを使い、提出コード用と評価関数用に別々に生成される
- ファクトリ入力は信頼済みの問題ファイルからだけ実行され、提出コード用と評価関数用に別々に生成される
- テストケースの `output.type` ごとに、`value` と `evaluator` のどちらで出力を判定するかが明確である
- 評価関数は `(output, input) => boolean` として実行され、`true` 以外は失敗になる
- テストケースがすべて開示される設計になっている
- バックエンドで採点しない責務分離になっている
- ソースコード保存先ディレクトリが起動時オプションで必須になっている
- 保存先ファイル衝突時に 429 を返し、フロントエンドが 1 秒後に再送する
- 課題セット一覧 API が単元ディレクトリからセット番号を返す
- 最新ソースコード取得 API が最新タイムスタンプのファイルを返し、対象なしの場合は 404 を返す
