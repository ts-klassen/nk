# 課題作成方針

このドキュメントは、各単元に対応する課題セットの作問方針とファイル構成を定める。

課題は `learned/NN_*.md` を知識範囲の契約として作る。単元 `NN` の課題では、`learned/NN_*.md` に書かれた知識と、それ以前の単元で学んだ知識だけを要求してよい。後続単元の知識や、教材外の JavaScript API、DOM、Node.js 固有の知識を暗黙に要求してはいけない。

作問時に、解答例や回答ソースコードをリポジトリ内へ生成してはいけない。課題として保存するのは、問題文 `.md` とテストケース `.json` だけにする。

## 課題セット

1 つの単元には、1 つ以上の課題セットを用意する。

各課題セットは、必ず次の 3 つの設問で構成する。

- `basic`: 基本
- `standard`: 標準
- `advanced`: 応用

保存先は次の形式にする。

```text
problems/${unit_number}/${set_index}/{basic,standard,advanced}.md
problems/${unit_number}/${set_index}/{basic,standard,advanced}.json
```

例:

```text
problems/03/1/basic.md
problems/03/1/basic.json
problems/03/1/standard.md
problems/03/1/standard.json
problems/03/1/advanced.md
problems/03/1/advanced.json
```

`unit_number` は `03` から `30` までの 2 桁文字列とする。`set_index` は単元ごとに `1` 始まりの連番とする。

## 難易度

難易度は、コード量だけでなく、問題文から実装手順を組み立てる距離で決める。

- 基本: 教材や `learned/NN_*.md` の内容をほぼそのまま再現する
- 標準: AtCoder Beginner Contest の A 問題程度の、1 段階の素直な実装にする
- 応用: AtCoder Beginner Contest の B 問題程度の、小さな処理設計が必要な実装にする

### 基本

教材に書かれていることをほぼそのまま書けば正解できる課題にする。

基本では、発想力や組み合わせよりも、当該単元で学んだ構文、メソッド、考え方をそのまま使えるかを確認する。

作問の目安:

- 教材本文または `learned/NN_*.md` の表現を、少しだけ変えた内容にする
- 入力値や変数名を変えても、本質的には教材の例と同じ書き方で解けるようにする
- 当該単元の新しい知識を 1 つだけ中心にする
- 以前の単元の知識は、変数宣言、式、条件分岐など必要最小限に留める

避けること:

- 複数の概念を同時に組み合わせないと解けない問題
- 教材にない API や実行環境の知識が必要な問題
- 入力の読み取りや出力形式でつまずく問題

### 標準

AtCoder Beginner Contest の A 問題程度の、短く素直な実装課題にする。

標準では、当該単元の主要な部分を理解していれば解ける問題にする。教材例の丸写しではなく、問題文から必要な処理を読み取り、当該単元の知識を使ってコードにできることを確認する。

作問の目安:

- 当該単元の主要知識を 1 つまたは強く関連する 2 つ程度使う
- 条件や入力値を読んで、短い処理に落とし込める問題にする
- 典型的なケースと境界値をテストケースに含める
- コード量は数行から十数行程度で解けるようにする

避けること:

- アルゴリズムの工夫が主題になる問題
- 先の単元で扱う構文や API を使わないと自然に解けない問題
- 採点がコードの書き方に依存しすぎる問題

### 応用

AtCoder Beginner Contest の B 問題程度の難易度を目安にする。

応用では、当該単元の知識だけを単独で使わせるのではなく、前の単元で学んだ知識と組み合わせて、問題文に合う処理を自分で構成できるかを確認する。標準よりも、問題文から処理手順へ分解する負荷を明確に高くする。

ABC B 問題相当とは、たとえば次のような処理を指す。ただし、各単元で未習の構文や API は使わせない。

- 条件を 1 回判定するだけでなく、複数の条件を組み合わせて結果を決める
- 候補の中から条件を満たすものを探す
- 条件を満たす要素を数える
- 文字列や配列を順に調べる
- 複数ステップで値や状態を更新する
- 問題文の仕様を、実装しやすい中間値や場合分けに落とし込む

作問の目安:

- 当該単元の知識を必ず解法の中心にする
- 以前の単元の知識を複数組み合わせる
- 問題文を読んで、処理手順、場合分け、中間値のいずれかを整理する必要がある内容にする
- 典型ケースだけでなく、境界値や例外的な分岐を意識しないと失敗する内容にする
- 解法として必要な処理は短く保つが、標準よりも 1 段階以上の処理の組み立てを要求する
- ループ、配列、文字列処理などは、その単元または以前の単元で扱っている場合だけ使わせる

避けること:

- 当該単元よりも以前の単元の復習が主題になる問題
- 未習のデータ構造、再帰、正規表現、非同期処理などを自然に必要とする問題
- 計算量の工夫や、競技プログラミングの典型テクニックを知らないと解けない問題
- 数学的発想が主題になり、JavaScript の学習課題としての焦点が弱くなる問題
- 実装量が長くなり、単元理解よりも作業量が主題になる問題
- テストケースだけから仕様を推測しないと解けない問題

## 問題文 `.md`

問題文は Markdown で書く。学習者が画面上で読み、そのままコードを書ける粒度にする。

推奨構成:

````markdown
# タイトル

## 問題

...

## 入力

`input` には ... が入っています。

## 出力

`output` に ... を代入してください。

## 例

`input`:

```json
3
```

期待される `output`:

```json
6
```
````

問題文では、標準入力や `console.log` を前提にしない。テスト実行時には必ず `input` という変数が与えられ、提出コードは `output` という変数へ結果を代入する。

テスト実行環境では、次の宣言が提出コードの前に存在する。

```js
const input = ...;
let output;
```

課題文では、`output` に代入すべき値の型を明記する。文字列、数値、真偽値、配列、オブジェクトなど、比較方法に影響するものは曖昧にしない。

## テストケース `.json`

テストケースは JSON で書き、フロントエンドから開示する。

基本形:

```json
{
  "title": "課題タイトル",
  "difficulty": "basic",
  "unit": "03",
  "set": 1,
  "cases": [
    {
      "name": "完全一致の代表例",
      "input": {
        "type": "value",
        "value": 3
      },
      "output": {
        "type": "value",
        "value": 6
      }
    }
  ]
}
```

フィールド:

- `title`: 問題文に表示する課題タイトル
- `difficulty`: `basic`、`standard`、`advanced` のいずれか
- `unit`: `03` から `30` までの 2 桁文字列
- `set`: 1 始まりの整数
- `cases`: テストケース配列
- `cases[].name`: テストケース名
- `cases[].input`: `input` 変数を生成するための指定
- `cases[].input.type`: `value`、`constructor`、`factory` のいずれか
- `cases[].input.value`: `input.type` が `value` の場合に指定する、`input` に入れる値
- `cases[].input.name`: `input.type` が `constructor` の場合に指定する、コンストラクタ名
- `cases[].input.args`: `input.type` が `constructor` の場合に指定する、コンストラクタへ渡す引数配列。省略時は空配列
- `cases[].input.factory`: `input.type` が `factory` の場合に指定する、`input` を返す関数の文字列
- `cases[].output`: `output` 変数の判定方法を指定する
- `cases[].output.type`: `value` または `evaluator`
- `cases[].output.value`: `output.type` が `value` の場合に指定する、期待する `output` の値
- `cases[].output.evaluator`: `output.type` が `evaluator` の場合に指定する、真偽値を返す評価関数の文字列

`input.type` が `value` のテストケースでは、`input.value` に指定した値を `input` 変数に入れる。`input.value` は JSON で表現できる値にする。

`input.type` が `constructor` のテストケースでは、`input.name` で指定したコンストラクタを `new` し、`input.args` を引数として渡した結果を `input` 変数に入れる。

```js
const input = new Date("2020-01-02T00:00:00.000Z");
```

`input.args` は JSON で表現できる値の配列にする。`input.args` を省略した場合は空配列として扱う。

コンストラクタ入力は、`Date`、`Map`、`Set` など、JSON 値だけでは表現しにくい入力を扱う場合に使う。使用できるコンストラクタは、当該単元または以前の単元で扱ったものに限る。

`input.type` が `factory` のテストケースでは、`input.factory` に指定した関数を実行し、その戻り値を `input` 変数に入れる。形式は、次のような JavaScript の関数式またはアロー関数式にする。

```js
() => inputValue
```

ファクトリ入力は、プレーンな JSON 値や単一のコンストラクタ呼び出しでは表現しにくい任意のオブジェクトを作る場合に使う。たとえば、メソッドを持つオブジェクト、複数のオブジェクトを関連づけた構造、入れ子になったインスタンスなどを入力にできる。

ファクトリ入力のケース例:

```json
{
  "name": "メソッドを持つオブジェクト",
  "input": {
    "type": "factory",
    "factory": "() => ({ count: 0, increment() { this.count += 1; return this.count; } })"
  },
  "output": {
    "type": "value",
    "value": 2
  }
}
```

コンストラクタ入力のケース例:

```json
{
  "name": "Date入力の年",
  "input": {
    "type": "constructor",
    "name": "Date",
    "args": ["2020-01-02T00:00:00.000Z"]
  },
  "output": {
    "type": "value",
    "value": 2020
  }
}
```

`output.type` が `value` のテストケースでは、実行後の `output` と `output.value` を完全一致で比較する。`output.value` は JSON で表現できる値にする。

`output.type` が `evaluator` のテストケースでは、実行後の `output` と元の `input` を評価関数に渡し、評価関数が `true` を返した場合だけ正解とする。評価関数に渡される `input` は、提出コードによる変更を受けていない値とする。

評価関数は JSON 内では文字列として保存する。形式は、次のような JavaScript の関数式またはアロー関数式にする。

```js
(output, input) => boolean
```

評価関数の例:

```json
{
  "name": "3以上10以下の数値",
  "input": {
    "type": "value",
    "value": { "min": 3, "max": 10 }
  },
  "output": {
    "type": "evaluator",
    "evaluator": "(output, input) => typeof output === \"number\" && input.min <= output && output <= input.max"
  }
}
```

`output.type` が `value` の `output.value` は JSON で表現できる値にする。`undefined`、関数、`Date`、`Map`、`Set` など JSON で直接表せない値を期待出力にしたい場合は、`output.type: "evaluator"` を使う。

`NaN`、`Infinity`、`-Infinity` は JSON で表せないため、`input.value`、`input.args`、`output.value` に使わない。

評価関数は、複数の正解が自然にある場合だけ使う。完全一致で十分に判定できる場合は `output.type: "value"` を使う。

評価関数で避けること:

- 現在日時、乱数、外部状態に依存する判定
- DOM、ネットワーク、ストレージなど実行環境に依存する判定
- 提出コードの書き方そのものを検査する判定
- 学習者に開示できない複雑な判定

ファクトリ入力で避けること:

- 現在日時、乱数、外部状態に依存する入力
- DOM、ネットワーク、ストレージなど実行環境に依存する入力
- 提出コードの書き方を検査するための入力
- 学習者に開示しても意味が伝わらない複雑な入力生成
- その単元までに扱っていない構文や API を理解しないと読めない入力生成

ファクトリ入力は、問題ファイルが信頼済みである前提で使う。外部ユーザーが投稿したテストケースや、信頼できない JSON から `input.factory` を実行してはいけない。

コンストラクタ入力で避けること:

- 未習のコンストラクタを使う入力
- コンストラクタ呼び出しの副作用に依存する入力
- 現在日時、乱数、外部状態に依存する入力
- 学習者に開示しても意味が伝わらない複雑な入力生成

## テストケース設計

各設問には 1 つ以上のテストケースを用意する。

テストケースはすべて学習者に開示される。そのため、隠しテストに頼らず、開示されたテストケースだけで仕様が十分に伝わるようにする。

テストケースには次を含める。

- 問題文の例と同じ代表ケース
- 条件分岐がある場合は、各分岐を通るケース
- 境界値がある場合は、境界の前後または境界そのもの
- 文字列や配列を扱う場合は、空文字列や空配列を含めるかを問題の意図に応じて決める
- 応用では、処理手順の一部だけを実装した解答では通らないケース
- 応用では、複数条件の優先順位、走査範囲、初期値、更新タイミングの誤りを検出できるケース
- 評価関数を使う場合は、正解すべき代表的な出力と、不正解にすべき出力を作問時に確認する

テストケースで避けること:

- 問題文に書いていない仕様を要求するケース
- 学習済みでない知識を使わないと対応できないケース
- 浮動小数点数の誤差で正誤が不安定になるケース
- 実行環境や現在日時に依存するケース

## 解答ファイルの禁止

作問 AI や作問者は、課題作成時に解答ソースコード、想定解、模範解答、解説用の実装ファイルをリポジトリ内に生成してはいけない。

禁止する例:

- `problems/03/1/basic-answer.js`
- `problems/03/1/solution.js`
- `answers/03/1/basic.js`
- `solutions/03/1/basic.js`
- `problems/03/1/README-solution.md`

作問時に正解例でテストケースを確認する必要がある場合でも、そのコードをリポジトリ内に保存しない。確認用コードが必要な場合は、作業中の一時的な実行だけに留め、課題ファイルとして残さない。

この禁止は、学習者がテスト実行時に提出したコードをバックエンドが保存する仕様とは別である。提出コードの保存先は採点システムの起動時オプションで指定されたディレクトリであり、作問時にリポジトリへ解答を置くことはしない。

## 品質チェック

作問後に次を確認する。

- `learned/NN_*.md` と、それ以前の `learned` ファイルだけで解ける
- 後続単元の知識を要求していない
- `basic`、`standard`、`advanced` の違いが明確である
- 各設問で `input` と `output` の型が明確である
- 各テストケースの `input.type` が `value`、`constructor`、`factory` のいずれかに統一され、必要なフィールドだけを持っている
- `input.type: "constructor"` を使う場合、そのコンストラクタが当該単元または以前の単元で扱われている
- `input.type: "factory"` を使う場合、そのファクトリが決定的で、当該単元または以前の単元の知識で読める
- 各テストケースの `output.type` が `value` または `evaluator` のどちらかに統一され、必要なフィールドだけを持っている
- テストケースがすべて開示されても問題として成立する
- すべてのテストケースに対し、課題仕様から導ける正解が通る
- 課題文と JSON の `title`、`difficulty`、`unit`、`set` が一致している
- 解答例、回答ソースコード、想定解、模範解答、解説用実装ファイルがリポジトリ内に生成されていない
