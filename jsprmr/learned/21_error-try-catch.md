# 21. 例外処理

## このページで学ぶこと

- `try...catch` 構文で例外が発生しうる処理と、例外発生時の処理を書ける
- `throw` 文で例外を投げ、`catch` 節の例外識別子から投げられた値を参照できる
- `Error` オブジェクトと代表的なビルトインエラーの役割を説明できる
- エラーログ、`name`/`message` プロパティ、スタックトレースをデバッグに利用できる
- `console.error` と Error Cause を使って、エラーの原因を追跡しやすくできる

## 課題で要求してよい知識

- `try` ブロック内で例外が発生すると、それ以降の `try` ブロック内の処理は実行されず、`catch` 節に処理が移ることを説明できる
- `catch` 節の例外識別子を使って、発生したエラーオブジェクトや投げられた値を参照できる
- `finally` 節が例外の発生有無に関係なく実行されることを説明できる
- `catch` 節または `finally` 節の片方だけでも `try` 文を書けるが、`finally` 節だけでは例外を処理できないことを説明できる
- `throw` 文と `new Error("メッセージ")` を使って、任意の条件で例外を投げられる
- `Error` オブジェクトの `message` プロパティから、作成時に渡したエラーメッセージを参照できる
- 文字列などではなく `Error` オブジェクトのインスタンスを例外として投げるべき理由を、スタックトレースの有無と関連づけて説明できる
- `ReferenceError` が存在しない変数や関数などの識別子を参照したときのエラーであることを説明できる
- `SyntaxError` が構文的に不正なコードを解釈しようとしたときのエラーであり、通常は実行前のパース段階で発生するため `try...catch` ではキャッチできないことを説明できる
- `TypeError` が期待される型ではない値を扱ったときのエラーであることを説明できる
- `Error` を継承したビルトインエラーのインスタンスを作成し、状況に応じて `throw` できる
- エラーオブジェクトの `name` プロパティと `message` プロパティから、エラーの種類と内容を読み取れる
- コンソールに表示されるエラーログから、エラーの種類、メッセージ、発生したファイル名、行番号、列番号を読み取れる
- スタックトレースを上から下へ読み、実際に例外が発生した場所と、その呼び出し元をたどれる
- `console.error` がメッセージとともにスタックトレースを出力し、エラーに関するログを `console.log` と区別できることを説明できる
- エラーをキャッチして別の `Error` を投げ直すと、本来のエラーのスタックトレースが失われる場合があることを説明できる
- `new Error("メッセージ", { cause: 元のエラー })` のように `cause` オプションを渡して、元のエラーの情報を引き継いだ新しいエラーを作成できる

## 前提として使う既習知識

- 変数宣言、条件分岐、関数定義、関数呼び出しを使える
- オブジェクトのプロパティを参照できる
- クラスやコンストラクタから `new` でインスタンスを作成できる
- `instanceof` で値が特定のコンストラクタのインスタンスかを判定できる
- `console.log` で値を出力できる

## 課題で要求しない知識（必要な場合のみ）

- `Error` を継承した独自エラークラスの定義は要求しない
- MDN のエラーリファレンスに載っているすべてのビルトインエラーの暗記は要求しない
- ブラウザや Node.js ごとの差異を含む、スタックトレース形式の完全な一致は要求しない
- `eval` 関数を使ったコード実行は、`SyntaxError` の性質を確認する用途を除いて要求しない
- 非同期処理における例外処理や Promise のエラーハンドリングは要求しない

## 課題で扱う場合の注意

- エラーメッセージやスタックトレースは実行環境によって差があるため、完全一致ではなくエラーの種類、原因、制御フローの説明を評価する
- `finally` 節を含む課題では、例外がキャッチされた場合とキャッチされない場合のどちらを扱うかを課題文で明示する
- ビルトインエラーを投げる課題では、`Error`、`TypeError` など、どのエラー型を使うべきかを状況から判断できる条件にする
- Error Cause を扱う課題では、`cause` に元のエラーオブジェクトを渡しているかを評価し、環境ごとの表示形式には依存しない
