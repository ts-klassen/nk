# 29. ECMAScript

## このページで学ぶこと

- JavaScriptの文法を定める仕様としてのECMAScriptと、その標準化を行うEcma InternationalおよびTC39の役割を学ぶ
- ECMAScriptのバージョンの歴史と、ES2015以降に毎年リリースされるようになった背景を学ぶ
- ECMAScriptがLiving Standardとして更新され、バージョンつき仕様が特定時点のスナップショットとして公開されることを学ぶ
- TC39のプロポーザル、チャンピオン、ステージによる仕様策定プロセスを学ぶ
- 策定中または新しい機能を試す手段としてのTranspilerとPolyfillの役割と限界を学ぶ
- ECMAScriptの仕様や策定プロセスが、言語学習、言語の進化の把握、情報の正しさの確認に役立つことを扱う

## 課題で要求してよい知識

- ECMAScriptがJavaScriptの文法を定める仕様であり、Ecma Internationalによって標準化されていることを説明できる
- TC39がECMAScript仕様を議論する技術委員会であり、ブラウザベンダーやECMAScriptに関心のある企業などで構成されることを説明できる
- ECMAScript 4が破棄されたこと、ES5.1からES2015までに時間がかかったこと、ES2015以降は毎年リリースされていることを、バージョンの歴史として説明できる
- Living Standardが、更新ごとにバージョン番号をつけず常に最新版を公開する仕様であることを説明できる
- Living StandardのECMAScriptと、ECMAScript 20XXのようなバージョンつき仕様が特定時点のドラフトを元にしたスナップショットであることを区別できる
- ブラウザなどのJavaScript実装が、基本的にLiving StandardのECMAScriptを参照している理由を説明できる
- ES2015以前の策定方式ではすべての合意後にリリースしていたため進化が停滞し、ES2016以降は毎年リリースするプロセスへ変更されたことを説明できる
- ECMAScriptに追加するAPIや構文などの機能が、個別のプロポーザルとして進められることを説明できる
- プロポーザルには責任者であるチャンピオンと、`0`、`1`、`2`、`2.7`、`3`、`4`のステージがあることを説明できる
- ステージ`0`がアイデア、`1`が機能提案、`2`が仕様書ドラフト作成、`2.7`が仕様完成後のテストやプロトタイプ検証、`3`が実装とフィードバック、`4`が仕様策定完了かつ2つ以上の実装がある段階であることを区別できる
- ステージ`2.7`が、ステージ`2`と`3`を行き来する問題を減らすため、実装前のテストと検証の段階として追加されたことを説明できる
- TC39のミーティングでプロポーザルごとにステージを進めるかを議論し、ステージ`4`になったプロポーザルがドラフト版へマージされ、毎年の決まった時期にECMAScript 20XXとしてリリースされる流れを説明できる
- `class`構文の例をもとに、合意できる最低限の機能を先に仕様へ入れ、追加機能を別プロポーザルとして進める考え方を説明できる
- ES4の失敗を背景として、プロポーザル単位で仕様策定を進めることにより、策定中断時の無駄を小さくしやすいことを説明できる
- 後方互換性を保てない場合や代替プロポーザルが出た場合など、プロポーザルが必ず仕様に入るわけではないことを説明できる
- 策定中のプロポーザルがJavaScriptエンジンに試験的なフラグつきで実装され、フラグを有効化すると試せる場合があることを説明できる
- Transpilerが、新しい構文を既存の機能で再現できるようにソースコードを変換するツールであることを説明できる
- Polyfillが、新しい関数やメソッドなどの仕様を満たす実装を提供するライブラリであることを説明できる
- 新しい構文の再現にはTranspiler、新しい関数やメソッドの提供にはPolyfillが関係することを、本文の例に沿って区別できる
- TranspilerやPolyfillは既存機能で新しい機能の再現を試みる手段であり、再現できない機能や完全には再現できない機能があることを説明できる
- 新しい機能を学ぶ目的でTranspilerやPolyfillに頼るべきではない理由を説明できる
- JavaScriptの言語機能の詳細を調べる先としてECMAScript仕様があり、使い方の確認ではMDN Web Docsのようなリファレンスを使う場面が多いことを説明できる
- ECMAScriptが後方互換性を尊重しながら進化しているため、今学んでいる内容が無駄になるわけではないことを説明できる
- プロポーザルは問題を解決するために提案されるため、機能が導入された理由や経緯を調べるときに策定プロセスを手がかりにできる
- ECMAScript仕様に入ったものは破壊的変更がほとんど行われず、ステージ`4`未満のプロポーザルはまだ安定していないと判断できる
- JavaScriptに関する情報の正しさを確認するときに、該当する仕様やプロポーザルの状態を確認する意義を説明できる
- JavaScriptという語がECMAScriptだけでなく、ウェブブラウザ、Node.js、WebAssembly、WebGL、WebRTCなど幅広い分野を含みうることを説明できる

## 前提として使う既習知識

- JavaScriptの基本文法、構文、API、関数、メソッド、ビルトインオブジェクトという語を区別できる
- `class`構文、`function`キーワード、配列のメソッドのような本文中の例を、既習のJavaScript機能として読める
- ブラウザやNode.jsのように、JavaScriptを実行する環境が複数あることを前提にできる
- GitHub、仕様書、リファレンス、リリース、ドラフトといった用語を、本文の説明範囲で読解できる

## 課題で要求しない知識（必要な場合のみ）

- ECMAScript仕様書のアルゴリズム表記や仕様本文を直接読んで解釈することは要求しない
- 現在のTC39プロポーザル一覧、各プロポーザルの最新ステージ、最新の議事録を外部サイトで調べることは要求しない
- Babel、TypeScript、core-jsをインストールまたは設定して実際に利用することは要求しない
- TranspilerやPolyfillの内部実装、変換後コード、ビルトインオブジェクトの書き換え方法は要求しない
- DOM API、HTML、CSS、WebAssembly、WebGL、WebRTC、Node.js固有APIの知識は要求しない
- TC39以外の標準化プロセスの詳細は要求しない

## 課題で扱う場合の注意

- このページは実装課題よりも説明・分類・判断の課題に向いている
- ECMAScriptやTC39プロポーザルの情報は変化するため、最新状況を問う課題では必要な資料や前提日時を課題文で与える
- プロポーザルのステージを扱う場合は、本文にあるステージ`0`、`1`、`2`、`2.7`、`3`、`4`の意味を基準に評価する
- TranspilerとPolyfillを扱う場合は、具体的なツール操作ではなく、役割の違いと限界を説明できるかを評価する
- バージョンの歴史を扱う場合は、細かな年表の暗記よりも、ES4の破棄、ES2015までの停滞、ES2015以降の毎年リリースという流れを中心に評価する
