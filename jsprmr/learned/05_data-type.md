# 05. データ型とリテラル

## このページで学ぶこと

- JavaScriptでは変数ではなく値にデータ型があり、データ型は大きくプリミティブ型とオブジェクトに分けられる
- `typeof`演算子で値のデータ型を調べられるが、オブジェクトの詳細な種類までは判定できない
- リテラルは、数値や文字列などの値をプログラム上へ直接書くための構文である
- 真偽値、数値、BigInt、文字列、`null`、オブジェクト、配列、正規表現の各リテラルを扱う
- プリミティブ型の値でも、文字列の`length`のようにプロパティを参照できる

## 課題で要求してよい知識

- JavaScriptは動的型付け言語であり、変数ではなく値にデータ型があることを説明できる
- データ型をプリミティブ型とオブジェクトに分類できる
- プリミティブ型の値は、一度作成した値そのものを変更できないイミュータブルな値であることを説明できる
- オブジェクトは、プリミティブ型やオブジェクトからなる集合であり、作成後も値自体を変更できるミュータブルな値であることを説明できる
- 7種類のプリミティブ型として、真偽値、数値、BigInt、文字列、`undefined`、`null`、シンボルを挙げられる
- プリミティブ型ではない値はオブジェクトであり、オブジェクト、配列、関数、クラス、正規表現、Dateなどがオブジェクトに分類されることを説明できる
- `typeof`演算子を使って、真偽値、数値、BigInt、文字列、シンボル、`undefined`、`null`、配列、オブジェクト、関数の評価結果を予測できる
- `typeof`では配列とオブジェクトのどちらも`"object"`になり、オブジェクトの詳細な種類を判定できないことを説明できる
- `typeof null`が`"object"`になることと、関数の`typeof`結果が`"function"`になることを説明できる
- リテラルが、データ型の値を直接記述するための構文であることを説明できる
- 真偽値、数値、BigInt、文字列、`null`、オブジェクト、配列、正規表現にリテラル表現があることを説明できる
- `true`と`false`の真偽値リテラルを書ける
- 数値リテラルには整数リテラルと浮動小数点数リテラルがあることを説明できる
- 数値リテラルがIEEE 754の倍精度浮動小数として扱われ、正確に扱える最大の整数が`2^53 - 1`であり、`Number.MAX_SAFE_INTEGER`で参照できることを説明できる
- 10進数、`0b`または`0B`から始まる2進数、`0o`または`0O`から始まる8進数、`0x`または`0X`から始まる16進数の整数リテラルを書ける
- 先頭が`0`の古い8進数表記は10進数と紛らわしく、避けるべき書き方であることを説明できる
- `.`を含む表記と`e`または`E`を含む表記で浮動小数点数リテラルを書ける
- `.123`のように先頭の`0`を省略できるが、読みやすさのため`0.123`のように書ける
- BigIntは任意精度の整数を扱うデータ型であり、数値の後ろに`n`を付けてBigIntリテラルを書ける
- BigIntリテラルに小数点を含めると構文エラーになることを説明できる
- Numeric Separatorsとして、数値リテラル内に`_`を入れて桁を読みやすくできる
- Numeric Separatorsの`_`は評価時に無視され、整数、浮動小数点、BigIntのリテラル内でのみ使えることを説明できる
- Numeric Separatorsの`_`をリテラルの先頭や数値の最後に置けないことを説明できる
- ダブルクォート、シングルクォート、バッククォートで文字列リテラルを書ける
- ダブルクォートとシングルクォートの文字列リテラルは同じ評価結果になることを説明できる
- 文字列リテラル内に同じクォート記号を含める場合は、バックスラッシュでエスケープするか、別のクォート記号で囲める
- ダブルクォートとシングルクォートの文字列リテラルでは、そのまま改行を書けず、改行には`\n`を使うことを説明できる
- テンプレートリテラルでは、バッククォートで囲んだ中に改行をそのまま書ける
- テンプレートリテラルで`${変数名}`を使い、変数の値を文字列へ埋め込める
- `null`は「値がない」ことを表す値であり、`null`リテラルで`null`値を書ける
- 未定義の変数参照が`ReferenceError`になる一方で、`null`を代入した変数は参照できることを説明できる
- `undefined`はリテラルではなくグローバル変数であり、`undefined`という名前の再定義は非推奨であることを説明できる
- `true`、`false`、`null`などのリテラルは変数名として使えず、再定義しようとすると構文エラーになることを説明できる
- `{}`のオブジェクトリテラルで新しいオブジェクトを作成できる
- オブジェクトリテラル内で、キーと値を`:`で区切ってプロパティを定義できる
- オブジェクトのプロパティ名と値を区別できる
- ドット記法とブラケット記法でオブジェクトのプロパティを参照できる
- ドット記法ではプロパティ名が識別子として書ける必要があり、数値から始まるようなプロパティ名はブラケット記法で参照することを説明できる
- `[]`の配列リテラルで空の配列を作成し、値をカンマ区切りで並べて要素を持つ配列を作成できる
- 配列は0から始まるインデックスに対応した値を保持し、`array[index]`で要素を参照できる
- `/パターン/`の正規表現リテラルで正規表現オブジェクトを作成できる
- 正規表現リテラルでは、`/`で囲んだパターン中の記号や特殊文字が意味を持つことを説明できる
- `new String("文字列")`のようなラッパーオブジェクトは`typeof`で`"object"`になり、文字列リテラルは`typeof`で`"string"`になることを区別できる
- 明示的にラッパーオブジェクトを作らなくても、プリミティブ型の文字列から`length`などのプロパティを参照できることを説明できる
- プリミティブ型のデータは、基本的にラッパーオブジェクトではなくリテラルで表現する方針を説明できる

## 前提として使う既習知識

- コメントを書ける
- `const`や`let`で変数を宣言し、値を代入できる
- `console.log`で値や評価結果を表示できる
- JavaScriptのコードを実行して、式の評価結果やエラーを確認できる

## 課題で要求しない知識（必要な場合のみ）

- オブジェクト、配列、関数、クラス、正規表現、Dateの詳細なAPIや内部仕様は要求しない
- 数値・BigIntの算術演算や比較演算の規則は要求しない
- 暗黙的な型変換やラッパーオブジェクトの詳しい仕組みは要求しない
- 正規表現パターンの詳細な読み書きや、`test`以外の正規表現メソッドは要求しない
- シンボルの作成以外の使い方や、一意性を利用した設計は要求しない
- プロトタイプ、クラス、メソッド定義、配列操作メソッドは要求しない

## 課題で扱う場合の注意

- このページの範囲では、オブジェクトと配列はリテラルで作成し、基本的なプロパティ参照やインデックス参照に留める
- 正規表現を扱う課題では、リテラルで正規表現オブジェクトを作れることを主眼にし、複雑なパターン作成を採点対象にしない
- `typeof null`や古い8進数表記など、仕様上の例外的な挙動は結果と理由の概略を評価し、詳細な歴史や仕様説明は要求しない
- エラーを扱う課題では、エラーメッセージの完全一致ではなく、`SyntaxError`や`ReferenceError`になる原因を説明できることを評価する
