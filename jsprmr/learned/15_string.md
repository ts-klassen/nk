# 15. 文字列

## このページで学ぶこと

- 文字列リテラル、テンプレートリテラル、エスケープシーケンスを使って文字列を作成する方法
- 文字列の結合、文字へのアクセス、長さ、比較、一部の取得、分解と結合などの基本操作
- JavaScript の文字列が UTF-16 の Code Unit の並びとして扱われること
- 文字列メソッドと正規表現を使った検索、マッチ、置換、削除の表現
- 構造を持つ文字列を安全に組み立てる考え方と、タグつきテンプレート関数の基本

## 課題で要求してよい知識

- `"`、`'`、`` ` `` の文字列リテラルで文字列を作成できる
- `"` と `'` の文字列リテラルに意味的な違いはなく、同じ内容なら `===` で同じ文字列として比較できることを説明できる
- 文字列リテラル中に同じクォート記号やバックスラッシュを含めるために、バックスラッシュでエスケープできる
- テンプレートリテラルでは改行をそのまま含められることを説明できる
- テンプレートリテラルの `${式}` で式の評価結果を文字列へ埋め込める
- `\n`、`\t`、`\\`、`\"`、`\'`、`` \` ``、`\uXXXX`、`\u{X}` などの代表的なエスケープシーケンスを使える
- 未定義のエスケープシーケンスではバックスラッシュが無視される場合があることを踏まえ、バックスラッシュ自体は `\\` と書ける
- `+` 演算子で文字列同士、または文字列と変数の値を結合できる
- `文字列[インデックス]` で指定位置の文字へアクセスし、存在しないインデックスでは `undefined` になることを予測できる
- `String.prototype.at` で指定位置の文字へアクセスし、負のインデックスで末尾からの位置を指定できる
- JavaScript の文字列は UTF-16 の Code Unit が順番に並んだものとして扱われることを説明できる
- `String.prototype.charCodeAt` で指定インデックスの Code Unit を整数として取得できる
- `String.fromCharCode` で Code Unit の整数値から文字列を作成できる
- `length` プロパティが文字列の Code Unit の個数を返し、空文字列では `0` になることを説明できる
- `===` で文字列の内容を比較し、Code Unit の並びと `length` が同じ場合に同じ文字列になることを説明できる
- `>`、`<`、`>=`、`<=` による文字列比較が、先頭から順に Code Unit の値を比較することを説明できる
- `split` で文字列を区切り文字や正規表現により配列へ分解できる
- 配列の `join` で配列の要素を区切り文字つきの文字列へ結合できる
- `split` と `join` を組み合わせて区切り文字を変換できる
- `slice` で開始位置から終了位置の直前までを取り出した新しい文字列を作成できる
- `slice` の第二引数を省略した場合、末尾までを取り出すことを説明できる
- `slice` に負の位置を指定すると末尾から数えた位置になり、開始位置が終了位置より大きい場合は空文字列を返すことを説明できる
- `substring` で開始位置から終了位置の直前までを取り出した新しい文字列を作成できる
- `substring` では負の位置が `0` として扱われ、開始位置が終了位置より大きい場合は引数が入れ替わることを説明できる
- `slice` や `substring` は元の文字列を変更せず、新しい文字列を返すことを説明できる
- `indexOf` や `lastIndexOf` で検索文字列が見つかったインデックスを取得し、見つからない場合は `-1` になることを説明できる
- `indexOf` で取得した位置を `slice` と組み合わせ、特定位置以降の文字列を取り出せる
- `startsWith`、`endsWith`、`includes` で検索文字列が先頭、末尾、または文字列中にあるかを真偽値で判定できる
- 正規表現オブジェクトがパターンとフラグで構成されることを説明できる
- 正規表現リテラル `/パターン/フラグ` と `RegExp` コンストラクタで正規表現オブジェクトを作成できる
- 正規表現の特殊文字 `\ ^ $ . * + ? ( ) [ ] { } |` が特別な意味を持つことを説明できる
- 正規表現内で特殊文字を文字として扱うためにバックスラッシュでエスケープできる
- `RegExp.escape` で文字列中の正規表現の特殊文字をエスケープできる
- 正規表現リテラルはコードのロード時にパターンが評価され、`RegExp` コンストラクタは実行時にパターンが評価されることを区別できる
- 実行時に変わるパターンを扱う場合は `RegExp` コンストラクタを利用できる
- `RegExp` コンストラクタへ渡すパターン文字列では、文字列リテラルとしてのバックスラッシュもエスケープする必要があることを説明できる
- `search` で正規表現にマッチした箇所のインデックスを取得し、マッチしない場合は `-1` になることを説明できる
- `match` で正規表現にマッチした文字列を取得し、マッチしない場合は `null` になることを説明できる
- `g` フラグなしの `match` が最初のマッチと `index`、`input` を含む特殊な配列を返すことを説明できる
- `g` フラグありの `match` がマッチしたすべての文字列を含む配列を返し、`index` と `input` は含まないことを説明できる
- `matchAll` に `g` フラグありの正規表現を渡し、`for...of` で各マッチの文字列、`index`、`input` を取り出せる
- 正規表現のキャプチャリングで、マッチした文字列全体とカッコで囲んだ部分を配列から取り出せる
- 繰り返しマッチするキャプチャ結果を取得する場合に `matchAll` を使える
- `RegExp.prototype.exec` の基本的な返り値が `match` に似た特殊な配列または `null` であることを説明できる
- `g` フラグありの `exec` では `lastIndex` が更新され、次回の検索開始位置に使われることを説明できる
- `RegExp.prototype.test` で正規表現にマッチするかを真偽値で判定できる
- `^`、`$`、通常のパターンと `test` を使い、先頭一致、末尾一致、含むかどうかを判定できる
- 固定文字列の検索では String メソッド、柔軟なパターン検索では正規表現を使うという判断ができる
- 正規表現を使う場合に、変数名やコメントで検索意図を補足できる
- 文字列はプリミティブ型で不変であり、文字列の一部を直接削除するのではなく新しい文字列を作ることを説明できる
- `replace` で検索文字列または正規表現にマッチする部分を別の文字列へ置換できる
- `replace` の第二引数に空文字列を指定して、削除に相当する新しい文字列を作成できる
- 文字列または `g` フラグなしの正規表現を指定した `replace` は最初に一致した部分だけを置換することを説明できる
- `g` フラグありの正規表現を指定した `replace` で、マッチしたすべての部分を置換できる
- `replaceAll` で検索文字列に一致するすべての部分を置換できる
- 正規表現では特殊な意味を持つ文字を、`replaceAll` の検索文字列としてそのまま扱える場合があることを説明できる
- `replace` や `replaceAll` の第二引数にコールバック関数を渡し、マッチ全体やキャプチャした文字列を使って置換文字列を組み立てられる
- URL やファイルパスのような構造を持つ文字列は、単純な `+` 結合だけでは意図しない文字列になる場合があることを説明できる
- 入力の表記揺れを吸収する専用の関数を作り、構造を持つ文字列を安全に組み立てられる
- タグつきテンプレート関数を `` 関数`テンプレート` `` の形式で呼び出せる
- タグ関数の第一引数に `${}` で区切られた文字列の配列、第二引数以降に `${}` の評価結果が順番に渡されることを説明できる
- タグ関数内で文字列の配列と値の配列を順番に結合し、テンプレート処理を実装できる
- `String.raw` をタグ関数として使える
- タグつきテンプレート関数で、埋め込む値に対して URL エスケープなどコンテキストに応じた処理を加えられる

## 前提として使う既習知識

- 変数宣言、関数定義、戻り値、条件分岐、ループを使える
- 数値、文字列、真偽値、`null`、`undefined` などの基本的な値を扱える
- 厳密等価演算子、関係演算子、論理演算子を使える
- オブジェクトのプロパティ、メソッド呼び出し、配列のインデックスアクセスを使える
- 配列の `slice`、`join`、`reduce`、`for...of` による反復処理を使える
- アロー関数、残余引数、分割代入を読める
- プリミティブ型の値が不変であることを前提にできる

## 課題で要求しない知識（必要な場合のみ）

- サロゲートペア、結合文字、絵文字、書記素クラスタ、Unicode 正規化など、次章以降で扱う Unicode の詳細は要求しない
- `Intl` やロケールに応じた文字列比較、照合順序の制御は要求しない
- URL オブジェクト、Node.js の `path` モジュール、DOM API、ネットワーク通信の知識は要求しない
- 正規表現の先読み、後読み、名前付きキャプチャ、後方参照など、このページで説明していない高度な正規表現構文は要求しない
- Iterator プロトコルや Generator の詳細な仕組みは要求しない
- HTML や DSL を実際にパースするタグつきテンプレート処理は要求しない
- strict mode における `delete` 演算子の詳細な仕様は要求しない

## 課題で扱う場合の注意

- `String.prototype.at` は ES2022、`matchAll` は ES2020、`replaceAll` は ES2021、`RegExp.escape` は ES2025 の機能として扱い、実行環境で利用できるかを課題文やテストで明確にする
- 文字列の `length`、インデックス、`slice`、`charCodeAt` は Code Unit 基準であるため、見た目の文字数を問う課題にしない
- Unicode の詳細を主題にしない課題では、絵文字や結合文字など Code Unit と見た目の文字がずれる入力を避ける
- 文字列メソッドと正規表現のどちらでも解ける課題では、どちらを使わせたいか、またはどちらも許可するかを明示する
- 正規表現を要求する課題では、使用してよい特殊文字やフラグを本文で扱った範囲に限定する
- `match` の返り値は `null` になる場合があるため、分割代入させる課題ではマッチする入力に限定するか、`null` の扱いも要求条件に含める
- 全件置換を扱う課題では、`replace` と `g` フラグつき正規表現を使うのか、`replaceAll` を使うのかを明示する
- 構造を持つ文字列の組み立てでは、実際の通信やファイルシステム操作ではなく、文字列としての組み立て結果を評価する
- 文字列の一部を `delete` できないことを扱う場合は、strict mode の詳細ではなく、文字列が不変であり新しい文字列を作る必要がある点を評価する
