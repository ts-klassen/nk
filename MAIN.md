# 見知らぬ言葉を API にする

Node.js / TypeScript / Express / MySQL で学ぶバックエンド入門

## はじめに

新卒研修の朝、あなたは社内のチャットを開く。

「この処理、冪等性が崩れてない？」

「このメトリクス、可観測性の観点で足りないかも」

「この用語、うちのチームだと別の意味で使ってるよ」

知っているようで知らない単語が並んでいる。辞書で調べると一般的な意味は出てくる。けれど、社内でその言葉がどんな場面で使われ、どんなニュアンスを持ち、どのチームでは少し違う意味で使われるのかは、辞書だけではわからない。

この教材では、最初に **書籍管理 API** を読みながらバックエンドの基本を学ぶ。次に、その考え方を使って **用例採集 API** を作る。

用例採集とは、言葉そのものにいきなり定義を与えるのではなく、その言葉が実際に使われた文、日時、補足を集めていくことだ。十分な用例が集まると、あとから社内向け用語集を作れる。最初から「正しい定義」を決めにいくのではなく、現場で言葉がどう使われているかを集める。

この教材のゴールは、Node.js の先端機能を覚えることではない。Express の流儀に詳しくなることでもない。REST API、CRUD、DB 連携、入力バリデーション、エラーハンドリング、Basic 認証、公開テストを通じて、バックエンド開発で何度も使う考え方を身につけることだ。

## 登場人物

この教材には、3 人の人物が登場する。

**美咲**  
新卒エンジニア。TypeScript はフロントエンド研修で触ったことがある。バックエンドと SQL はまだ慣れていない。

**陸**  
同期。動けばよい実装を早く書けるが、HTTP ステータスコードやエラー形式はまだ曖昧。

**佐伯**  
メンター。Erlang、Go、Perl、Rust のバックエンドを見てきた。Node.js には詳しすぎる必要はないが、API と DB の約束には厳しい。

佐伯は最初にこう言う。

> 言語は変わる。けれど、API が約束であること、DB が状態を持つこと、エラーが仕様であることは変わらない。

この教材は、その約束を一つずつ作っていく。

## 技術条件

この教材では、次の技術を使う。

- Node.js 22 以上
- TypeScript
- `tsc` によるビルド
- Express v5
- MySQL
- Docker Compose
- `mysql2`
- `express-validator`
- `argon2`
- Mocha / Chai

使わないものも重要である。

- フロントエンドは作らない
- フロントエンドと型やコードを共有しない
- ORM は使わない
- migration ツールは使わない
- JWT は使わない
- セッション管理はしない
- 動的 HTML は返さない

### コラム: なぜフロントエンドと型を共有しないのか

TypeScript を使っていると、フロントエンドとバックエンドで型を共有したくなる。便利だからだ。

しかし、この研修では共有しない。実際の開発では、バックエンドが Erlang、Go、Perl、Rust で書かれていることがある。言語が違えば、TypeScript の型をそのまま共有できない。

大切なのは、コードを共有することではない。**API 仕様という約束を共有すること**である。

フロントエンドとバックエンドは、同じファイルを見るのではなく、同じ仕様を守る。ここではその感覚を身につける。

## System A と System B の対応

2 つのシステムは、似ているが完全には同じではない。

| 学習対象 | System A: 書籍管理 | System B: 用例採集 |
| --- | --- | --- |
| 親リソース | 書籍 | 用語 |
| 子リソース | 読書メモ | 用例 |
| ユーザー | 認証用ユーザー | 認証用ユーザー |
| 親リソース CRUD | 認証不要 | 認証不要 |
| 子リソース作成 | 認証必須 | 認証必須 |
| 子リソース閲覧 | 本人だけ | 誰でも |
| 子リソース更新・削除 | 本人だけ | 所有者だけ |
| 親削除時 | 子があれば `409` | 子があれば `409` |

System A では、読書メモは本人だけが見られる。System B では、用例は誰でも見られる。ただし編集と削除は所有者だけだ。

この差分が、課題の大事な部分である。単に名前を置き換えるだけでは足りない。公開範囲と所有者チェックの違いを理解する必要がある。

## 第 1 章 API は約束である

美咲は最初に `GET /books` を見て、こう言う。

> 本の一覧を返すだけなら、配列を返せばよさそうです。

佐伯は首を振る。

> 返せばよい、では足りない。どう返すかが約束になる。

REST API は、HTTP を使った約束である。URL はリソースを表し、HTTP メソッドは操作を表す。

| 操作 | HTTP method | System A | System B |
| --- | --- | --- | --- |
| 一覧取得 | `GET` | `GET /books` | `GET /terms` |
| 詳細取得 | `GET` | `GET /books/:bookId` | `GET /terms/:termId` |
| 作成 | `POST` | `POST /books` | `POST /terms` |
| 更新 | `PATCH` | `PATCH /books/:bookId` | `PATCH /terms/:termId` |
| 削除 | `DELETE` | `DELETE /books/:bookId` | `DELETE /terms/:termId` |

成功時のステータスコードも約束である。

| 操作 | status |
| --- | --- |
| `GET` | `200 OK` |
| `POST` | `201 Created` |
| `PATCH` | `204 No Content` |
| `DELETE` | `204 No Content` |

ここで重要なのは、`PATCH` と `DELETE` は成功時にレスポンスボディを返さないことだ。`204 No Content` は「成功した。返す本文はない」という意味を持つ。

### System A で学ぶ

`docs/system-a/api.md` を開き、書籍 API を確認する。

見る場所:

- `GET /books`
- `POST /books`
- `GET /books/:bookId`
- `PATCH /books/:bookId`
- `DELETE /books/:bookId`

次に [system-a/src/app.ts](system-a/src/app.ts) を開き、対応する Express ルートを探す。

探すもの:

- `app.get("/books", ...)`
- `app.post("/books", ...)`
- `app.get("/books/:bookId", ...)`
- `app.patch("/books/:bookId", ...)`
- `app.delete("/books/:bookId", ...)`

仕様書に書かれた HTTP メソッドと URL が、コード上のルート定義に対応していることを確認する。

### System B で演習する

`docs/system-b/api.md` を開き、用語 API を確認する。

演習:

1. `GET /terms` に対応する Express ルート名を書く。
2. `POST /terms` の成功ステータスコードを書く。
3. `PATCH /terms/:termId` が成功したとき、レスポンスボディを返すべきか答える。
4. `DELETE /terms/:termId` の成功ステータスコードを書く。

答えは実装ではなく、まず仕様書から導く。仕様から導けないものは、実装してはいけない。

### コラム: 認証不要で削除できる「変な仕様」

この教材では、書籍や用語は認証なしで削除できる。普通の業務システムなら危ない仕様である。

なぜそんな仕様にするのか。理由は、学習の焦点を分けるためだ。

書籍や用語では CRUD と DB 連携を学ぶ。読書メモや用例では Basic 認証と所有者チェックを学ぶ。全部に認証を入れると、初学者は「CRUD で詰まっているのか」「認証で詰まっているのか」を切り分けにくくなる。

変な仕様にも、教材上の役割がある。

## 第 2 章 JSON とエラーの形をそろえる

陸は最初の実装で、見つからない本に対してこう返した。

```json
{
  "message": "not found"
}
```

別の API ではこう返した。

```json
{
  "error": "missing"
}
```

佐伯は言う。

> 人間にはどちらも読める。でも API の利用者には、違う形のエラーが増えるほど扱いづらくなる。

この教材では、エラー形式を統一する。

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "対象リソースが存在しません"
  }
}
```

使う主なエラーコードは次の通り。

| status | code | message |
| --- | --- | --- |
| `400` | `VALIDATION_ERROR` | `入力値が不正です` |
| `401` | `UNAUTHORIZED` | `認証が必要です` |
| `403` | `FORBIDDEN` | `この操作は許可されていません` |
| `404` | `NOT_FOUND` | `対象リソースが存在しません` |
| `409` | `CONFLICT` | `リソースが競合しています` |
| `500` | `INTERNAL_SERVER_ERROR` | `サーバーエラーが発生しました` |

`500 INTERNAL_SERVER_ERROR` は想定外エラー用である。通常の課題実装で意図的に発生させる対象ではない。

### System A で学ぶ

見るファイル:

- [system-a/src/errors.ts](system-a/src/errors.ts)
- [system-a/src/app.ts](system-a/src/app.ts)

確認すること:

1. `AppError` が `status`、`code`、`message` を持つこと。
2. `errorHandler` が `AppError` を共通 JSON 形式に変換していること。
3. `findBook()` が存在しない書籍を `404 NOT_FOUND` にしていること。
4. ISBN 重複を `409 CONFLICT` に変換していること。

### System B で演習する

用例採集 API でも同じエラー形式を使う。

演習:

1. 存在しない用語 ID を指定されたとき、どの status / code / message を返すか書く。
2. `term` が重複したとき、どの status / code / message を返すか書く。
3. 認証なしで `POST /terms/:termId/examples` を呼ばれたとき、どの status / code / message を返すか書く。

ここでの目的は、エラーを「実装の都合」ではなく「API の仕様」として扱うことである。

## 第 3 章 DB は状態を持つ

API はリクエストを受けてレスポンスを返す。しかしバックエンドは、それだけではない。状態を持つ。

書籍を作成したら、次の一覧取得でその書籍が返ってくる。用語を更新したら、次の詳細取得で更新後の値が返ってくる。この「次にも残るもの」を DB が担う。

この教材では MySQL を使う。ORM は使わず、SQL を直接書く。

### なぜ ORM を使わないのか

ORM は便利である。しかし、初めてバックエンドを学ぶ段階では、便利さが見えにくさにもなる。

この教材では、API と SQL の対応を見えるようにする。

- 一覧取得では `SELECT`
- 作成では `INSERT`
- 更新では `UPDATE`
- 削除では `DELETE`
- 件数取得では `COUNT(*)`

SQL の高度な学習は別枠で行う。ここでは、API が DB の状態をどう読み書きするかに集中する。

### System A で学ぶ

見るファイル:

- [system-a/sql/schema.sql](system-a/sql/schema.sql)
- [system-a/src/db.ts](system-a/src/db.ts)
- [scripts/reset-db.mjs](scripts/reset-db.mjs)

確認すること:

1. `books` テーブルに `isbn`、`title`、`author`、`published_date` がある。
2. `isbn` に一意制約がある。
3. `reading_notes` が `user_id` と `book_id` を持つ。
4. `ON DELETE CASCADE` を使っていない。
5. `scripts/reset-db.mjs` が DB を作り直して `schema.sql` を実行している。

### System B で演習する

見るファイル:

- [system-b/sql/schema.sql](system-b/sql/schema.sql)

演習:

1. `terms` テーブルの一意制約を確認する。
2. `examples` テーブルが、どのカラムで所有者を表しているか答える。
3. `examples` テーブルが、どのカラムで用語との紐付きを表しているか答える。
4. 用語削除時に用例を連動削除しない理由を説明する。

### コラム: 親を消したら子も消すべきか

書籍に読書メモがある。用語に用例がある。このとき親を削除したら、子も一緒に消すべきだろうか。

この教材では消さない。子データがある場合、親の削除は `409 CONFLICT` にする。

理由は単純だ。読書メモや用例は、ユーザーが時間を使って作ったデータである。親を消しただけで一緒に消えると、何が起きたのか見えにくい。ここでは「今の状態では削除できない」と明示する。

## 第 4 章 一覧 API は配列を返すだけではない

美咲は `GET /books` のレスポンスを見て、少し驚く。

```json
{
  "items": [],
  "pagination": {
    "limit": 20,
    "offset": 0,
    "total": 0
  }
}
```

> 配列だけじゃないんですね。

佐伯は答える。

> 画面や別システムが次に何をすればよいか判断するには、件数とページング情報が必要になる。

一覧 API では `limit` と `offset` を使う。

- `limit`: 何件返すか。デフォルト `20`、最大 `100`
- `offset`: 何件飛ばすか。デフォルト `0`
- `total`: 条件に合う総件数
- 並び順: `id ASC`

### System A で学ぶ

`GET /books` の実装を見る。

確認すること:

1. `validatePagination()` で `limit` と `offset` を検証している。
2. `getPagination()` でデフォルト値を決めている。
3. `COUNT(*)` で `total` を取得している。
4. `ORDER BY id ASC LIMIT ? OFFSET ?` で一覧を取得している。

### System B で演習する

用例採集では、次の 2 つが一覧 API である。

- `GET /terms`
- `GET /terms/:termId/examples`

演習:

1. `GET /terms?limit=1&offset=0` が `pagination.limit = 1` を返すようにする。
2. 同じ用語に用例を 2 件作り、`GET /terms/:termId/examples?limit=1&offset=1` が 2 件目だけを返すようにする。
3. `limit=101` は `400 VALIDATION_ERROR` にする。

公開テストは、用語一覧だけでなく用例一覧のページングも確認する。親リソースだけページングできても、子リソース一覧が未対応なら仕様を満たしていない。

## 第 5 章 入力値は信用しない

API の利用者は、いつも正しい JSON を送ってくれるとは限らない。

`publishedDate` に `2026-99-99` が来るかもしれない。`limit` に `9999` が来るかもしれない。`PATCH` の body に `null` が来るかもしれない。

入力値を信用しない。これはバックエンドの基本である。

### System A で学ぶ

見るファイル:

- [system-a/src/validation.ts](system-a/src/validation.ts)
- [system-a/src/app.ts](system-a/src/app.ts)

確認すること:

1. `validateId()` が path parameter を正整数にしている。
2. `validatePagination()` が `limit` と `offset` を検証している。
3. `requireAtLeastOne()` が空の PATCH を `400` にしている。
4. `requireAtLeastOne()` が JSON `null` や配列も `400` にしている。
5. `publishedDateRule()` が実在する日付だけを通す。

### System B で演習する

用例採集で検証すべき入力:

- `username`: 半角英数字と `_`、3 文字以上 32 文字以下
- `password`: 8 文字以上 72 文字以下
- `term`: 1 文字以上 255 文字以下
- `body`: 1 文字以上 2000 文字以下
- `collectedDate`: 実在する日付の `YYYY-MM-DD`
- `note`: 任意、1000 文字以下
- `limit`: 1 以上 100 以下
- `offset`: 0 以上

演習:

1. `POST /terms` に `{ "term": "" }` を送ったら `400 VALIDATION_ERROR` にする。
2. `GET /terms?limit=101` を `400 VALIDATION_ERROR` にする。
3. `PATCH /examples/:exampleId` に JSON `null` を送ったら `400 VALIDATION_ERROR` にする。

### コラム: DB に怒られる前に API が断る

不正な入力を DB まで流すと、DB がエラーを返すことがある。しかし、そのエラーをそのまま `500` にすると、API の利用者には「サーバーが壊れた」ように見える。

入力不正はサーバーの故障ではない。利用者が直せるリクエストの問題である。だから `400 VALIDATION_ERROR` にする。

## 第 6 章 認証と認可は違う

認証と認可は似た言葉だが、別のものだ。

- 認証: あなたは誰か
- 認可: あなたはそれをしてよいか

Basic 認証で `username` と `password` を検証できたら、認証は成功である。しかし、認証できたからといって、他人の読書メモや用例を編集してよいわけではない。そこで認可が必要になる。

### Basic 認証

この教材では JWT もセッションも使わない。Basic 認証を使う。

```http
Authorization: Basic base64(username:password)
```

認証に失敗した場合は、次のヘッダを返す。

```http
WWW-Authenticate: Basic realm="backend-training"
```

パスワードは平文保存しない。Argon2 でハッシュ化して `password_hash` に保存する。Argon2 のパラメータ調整は扱わず、ライブラリのデフォルトパラメータを使う。

### System A で学ぶ

見るファイル:

- [system-a/src/auth.ts](system-a/src/auth.ts)
- [system-a/src/app.ts](system-a/src/app.ts)

確認すること:

1. `parseBasicAuth()` が `Authorization` ヘッダを読む。
2. `users` テーブルから `username` でユーザーを探す。
3. `argon2.verify()` でパスワードを検証する。
4. 成功したら `req.user` にユーザー情報を入れる。
5. 失敗したら `401 UNAUTHORIZED` と `WWW-Authenticate` を返す。

### System B で演習する

公開テストでは、ユーザー登録後に DB の `password_hash` を直接確認する。

満たすべき条件:

1. `password_hash` は平文パスワードと違う。
2. `$argon2` で始まる。
3. `argon2.verify()` で元のパスワードを検証できる。

つまり、レスポンスにパスワードを出さないだけでは足りない。DB に平文保存していないことまで仕様である。

## 第 7 章 所有者だけができる操作

読書メモと用例は、どちらもユーザーが作る子リソースである。しかし公開範囲が違う。

System A の読書メモは本人だけが見られる。System B の用例は誰でも見られる。

ただし、どちらも更新と削除は所有者だけができる。

| 操作 | 読書メモ | 用例 |
| --- | --- | --- |
| 作成 | 認証必須 | 認証必須 |
| 一覧 | 本人だけ | 誰でも |
| 詳細 | 本人だけ | 誰でも |
| 更新 | 所有者だけ | 所有者だけ |
| 削除 | 所有者だけ | 所有者だけ |

### System A で学ぶ

見るコード:

- `GET /books/:bookId/reading-notes`
- `POST /books/:bookId/reading-notes`
- `GET /reading-notes/:noteId`
- `PATCH /reading-notes/:noteId`
- `DELETE /reading-notes/:noteId`

確認すること:

1. 一覧取得では `WHERE book_id = ? AND user_id = ?` で自分のメモだけ返す。
2. 詳細、更新、削除では `assertOwnReadingNote()` で所有者チェックをする。
3. 他ユーザーの読書メモなら `403 FORBIDDEN` を返す。

### System B で演習する

用例は誰でも見られる。

- `GET /terms/:termId/examples` は認証不要
- `GET /examples/:exampleId` は認証不要

しかし、作成、更新、削除は別である。

- `POST /terms/:termId/examples` は認証必須
- `PATCH /examples/:exampleId` は認証必須、所有者だけ
- `DELETE /examples/:exampleId` は認証必須、所有者だけ

演習:

1. 他ユーザーの用例を `GET /examples/:exampleId` で取得できるようにする。
2. 他ユーザーの用例を `PATCH` したら `403 FORBIDDEN` にする。
3. 他ユーザーの用例を `DELETE` したら `403 FORBIDDEN` にする。
4. 認証なしで `PATCH` または `DELETE` したら `401 UNAUTHORIZED` にする。

### コラム: `401` と `403`

`401` は「あなたが誰かわからない」である。Basic 認証がない、または間違っている。

`403` は「あなたが誰かはわかった。でもその操作はできない」である。他人の用例を編集しようとしている。

この違いを曖昧にすると、API の利用者は何を直せばよいかわからなくなる。

## 第 8 章 用例採集とは何か

美咲は用例採集 API の仕様を読んで、不思議に思う。

> 用語に description がないんですね。説明を書かないんですか？

佐伯は答える。

> 最初から説明を書くと、言葉の使われ方を狭めることがある。まずは用例を集める。

用例採集では、用語そのものには説明文を持たせない。

```json
{
  "term": "冪等性"
}
```

用例には、実際に見聞きした文、採集日、備考を持たせる。

```json
{
  "body": "PUT は冪等な操作として設計することが多い。",
  "collectedDate": "2026-04-28",
  "note": "API設計レビューでの発言"
}
```

なぜこうするのか。

言葉は、定義だけでは生きていない。誰が、どの場面で、何を問題にしてその言葉を使ったのかによって、理解が変わる。

社内でしか使われない言葉は、特にそうだ。

- プロジェクト固有の略語
- 歴史的経緯で残った呼び名
- チーム内だけで通じる運用用語
- 一般用語だが社内では少し違う意味の言葉

こうした言葉にいきなり定義を与えると、現場の使われ方を取り逃がすことがある。だからまず、用例を集める。

### ストーリー: 小さな用語集ができるまで

陸は最初、「冪等性」という用語を 1 行で説明しようとした。

> 同じ操作を何度実行しても結果が同じこと。

間違いではない。しかしレビューでは、別の文脈が出てきた。

「この POST は冪等キーがあるから再送できる」

「PUT は冪等に設計したい」

「このバッチ、途中で落ちても冪等に再実行できる？」

同じ言葉でも、API、バッチ、リトライ、障害対応で少しずつ使われ方が違う。用例を 10 件、20 件と集めると、ようやく社内向けの説明が書ける。

用例採集 API は、その最初の箱である。

## 第 9 章 公開テストは仕様の読者である

System B には非公開テストがない。すべてのテストが受講生に見える。

これは「テストを見れば答えがわかる」という意味ではない。むしろ逆である。公開テストは、仕様をどの程度満たしているかを受講生自身が確認するための道具である。

見るファイル:

- [system-b/test/system-b.test.mjs](system-b/test/system-b.test.mjs)

公開テストが確認する主なもの:

- ユーザー登録
- Argon2 によるパスワードハッシュ保存
- Basic 認証失敗時の `401` と `WWW-Authenticate`
- 用語 CRUD
- 用語一覧の `limit` / `offset` / `total`
- 用例作成
- 用例一覧の `limit` / `offset` / `total`
- 用例詳細の公開閲覧
- 用例の所有者更新・削除
- 他ユーザー更新・削除時の `403`
- 子データあり用語削除時の `409`
- 存在しないリソースの `404`
- 入力エラーの `400`

### 演習

System B を実装する前に、公開テストを読む。

1. テスト名だけを上から読む。
2. どの API を実装すればよいかメモする。
3. API 仕様書の該当箇所へ戻る。
4. テストが見ている値と、仕様書の値が一致しているか確認する。

テストは仕様の代わりではない。仕様を確認する読者である。

## 第 10 章 System B を実装する順序

最後に、用例採集 API を実装する。

おすすめの順序は次の通り。

1. プロジェクトの起動
2. MySQL 接続
3. 共通エラー処理
4. 入力バリデーション
5. `POST /users`
6. Basic 認証
7. 用語 CRUD
8. 用語一覧のページング
9. 用例作成
10. 用例一覧と詳細取得
11. 用例更新と削除
12. 所有者チェック
13. 子データあり用語削除時の `409`
14. 公開テストを実行し、仕様との差分を直す

### 最初にやること

```bash
npm install
npm run db:up
npm run db:reset:b
npm run test:b
```

最初の `npm run test:b` は失敗する。まだ実装していないからだ。

大切なのは、失敗を読むことだ。

- どの API が呼ばれているか
- 期待ステータスコードは何か
- 期待レスポンスは何か
- DB に何が保存されているべきか

失敗は作業指示である。

### 実装中の心得

1. 先に API 仕様を見る。
2. 次に公開テストを見る。
3. それから実装する。
4. 失敗したら、エラーメッセージを読む。
5. 実装を直す前に、仕様をもう一度見る。

陸は途中でこう言う。

> テストを通すだけなら、雑に分岐を書いてもよさそうです。

佐伯は答える。

> それは公開テストを相手にしている。相手にするのは仕様だ。

テストは仕様の一部を確認する。仕様そのものではない。公開テストにないから実装しなくてよい、とは考えない。

## 最後に

この教材で作るものは、小さな API である。

しかし、その中にはバックエンド開発の基本が詰まっている。

- API は約束である。
- DB は状態を持つ。
- 入力値は信用しない。
- エラーは仕様である。
- 認証と認可は違う。
- テストは仕様の読者である。
- 実装は、仕様をコードに翻訳する作業である。

用例採集 API が完成したら、社内の見知らぬ言葉を 1 つ登録してみる。

用語だけを登録する。まだ説明は書かない。次に、その言葉が実際に使われた文を 1 つ採集する。採集日と備考も残す。

数が集まると、ただの単語リストではなくなる。現場の会話、レビューの温度、障害対応の記憶、チームごとの言葉の癖が見えてくる。

バックエンド API は、ただデータを保存する箱ではない。誰かが後で理解できる形で、出来事を残すための約束でもある。
