# 全体の整合性を確認する

## 目的

教材、システム A、システム B、API 仕様、DB スキーマ、テストの整合性を確認する。

## 確認観点

- Node.js の先端機能や Express 固有知識に寄りすぎていないか。
- REST API、CRUD、DB 連携が中心になっているか。
- TypeScript の説明が既習内容と重複しすぎていないか。
- SQL の説明が別枠学習と重複しすぎていないか。
- フロントエンドとの型共有やコード共有が入っていないか。
- API 仕様書が Markdown で厳密に書かれているか。
- エラーレスポンス形式が一貫しているか。
- `VALIDATION_ERROR`, `UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `CONFLICT` の使い分けが一貫しているか。
- Basic 認証と `WWW-Authenticate` ヘッダが仕様・実装・テストに入っているか。
- 書籍管理と用例採集の対応関係が崩れていないか。
- 読書メモは本人だけが見られる仕様になっているか。
- 用例は誰でも見られ、編集削除は所有者だけになっているか。
- 書籍・用語の CRUD が認証不要になっているか。
- 書籍・用語削除時、子データありの場合に `409 Conflict` になっているか。
- システム B のテストが全て公開されているか。
- 非公開テストや受講生向けセルフチェックリストを作っていないか。
- 評価基準が課題文に漏れていないか。
